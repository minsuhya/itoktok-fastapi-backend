# ITokTok 데이터베이스 관리 가이드

> 최종 수정일: 2026-04-01

---

## 1. 개요

ITokTok은 외부 MySQL/MariaDB 서버를 주 데이터베이스로 사용하며, SQLModel ORM과 Alembic 마이그레이션을 통해 스키마를 관리한다. 이 문서는 일상적인 DB 운영부터 스키마 변경, 트러블슈팅까지의 실무 가이드를 제공한다.

### 기술 스택

| 구성 요소 | 기술 | 비고 |
|-----------|------|------|
| DBMS | MySQL / MariaDB | AWS RDS (외부 서버) |
| ORM | SQLModel (SQLAlchemy 기반) | `backend/app/models/` |
| Migration | Alembic | `backend/alembic/` |
| 연결 | `CONN_URL` 환경변수 | `.env` 파일에서 로드 |
| Python 환경 | `.venv` (uv, Python 3.11) | `backend/.venv/` |

## 2. 환경 설정

### 2.1 DB 연결 정보

루트 디렉토리의 `.env` 파일에 설정한다:

```bash
CONN_URL="mysql+pymysql://user:password@host:port/database?charset=utf8mb4"
```

- 비밀번호에 `@` 등 특수문자가 포함되면 URL 인코딩 필요 (예: `@` → `%40`)
- `.env` 파일은 `.gitignore`에 포함되어야 한다

### 2.2 로컬 실행 환경

```bash
cd backend

# 가상 환경의 Python 사용
.venv/bin/python -c "import sqlmodel; print('OK')"

# Alembic 실행
.venv/bin/alembic current
```

> Alembic의 `env.py`가 `dotenv`로 `.env`를 자동 로드하므로 별도 환경변수 export 불필요.

## 3. 모델(테이블) 관리

### 3.1 현재 테이블 구조

```
backend/app/models/
  user.py          → User, CenterInfo, UserSearchSelectedTeacher
  client.py        → ClientInfo
  schedule.py      → Schedule, ScheduleList
  program.py       → Program
  record.py        → Record
  voucher.py       → Voucher
  announcement.py  → Announcement
  inquiry.py       → Inquiry
```

### 3.2 모델 작성 규칙

**관계를 가진 모델은 반드시 같은 파일에 정의한다.**

```python
# user.py — User와 CenterInfo는 관계가 있으므로 같은 파일에 둔다
class User(SQLModel, table=True):
    center_username: str = Field(foreign_key="centerinfo.username")
    center_info: Optional["CenterInfo"] = Relationship(back_populates="users")

class CenterInfo(SQLModel, table=True):
    users: List["User"] = Relationship(back_populates="center_info")
```

순환 import를 방지하기 위해 `TYPE_CHECKING` 블록을 활용한다:

```python
from typing import TYPE_CHECKING

if TYPE_CHECKING:
    from .client import ClientInfo
    from .schedule import Schedule
```

### 3.3 필수 컬럼 패턴

모든 핵심 테이블은 다음 컬럼을 포함한다:

```python
class MyModel(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    # ... 비즈니스 필드 ...
    created_at: Optional[datetime] = Field(
        sa_column=Column(DateTime(timezone=True), server_default=func.now())
    )
    updated_at: Optional[datetime] = Field(
        sa_column=Column(DateTime(timezone=True), onupdate=func.now())
    )
    deleted_at: Optional[datetime] = Field(default=None)  # Soft Delete
```

## 4. Alembic 마이그레이션

### 4.1 기본 명령어

```bash
cd backend

# 현재 DB 버전 확인
.venv/bin/alembic current

# 마이그레이션 히스토리
.venv/bin/alembic history

# 모델 변경 후 migration 자동 생성
.venv/bin/alembic revision --autogenerate -m "설명"

# 최신으로 업그레이드
.venv/bin/alembic upgrade head

# 한 단계 롤백
.venv/bin/alembic downgrade -1

# 특정 revision으로 롤백
.venv/bin/alembic downgrade <revision_id>

# DB 변경 없이 버전만 마킹 (baseline 등)
.venv/bin/alembic stamp head
```

### 4.2 스키마 변경 워크플로우

```
1. 모델 수정 (app/models/*.py)
       ↓
2. migration 자동 생성
   .venv/bin/alembic revision --autogenerate -m "add_column_to_user"
       ↓
3. 생성된 파일 검토 (backend/alembic/versions/<hash>_*.py)
   - upgrade()와 downgrade()가 의도한 변경인지 확인
   - autogenerate가 감지하지 못하는 변경 수동 추가
       ↓
4. 개발 환경에서 적용 및 테스트
   .venv/bin/alembic upgrade head
       ↓
5. 프로덕션 적용 전 DB 백업
   mysqldump -h <host> -u <user> -p <database> > backup_$(date +%Y%m%d).sql
       ↓
6. 프로덕션 적용
   .venv/bin/alembic upgrade head
       ↓
7. 앱 정상 동작 확인
```

### 4.3 새 모델 추가 시 체크리스트

1. `backend/app/models/` 에 모델 파일 생성
2. `backend/alembic/env.py` 에 모델 import 추가 **(필수!)**
3. `backend/app/schemas/` 에 Pydantic 스키마 추가
4. `backend/app/crud/` 에 CRUD 함수 추가
5. `backend/app/api/` 에 API 라우트 추가
6. `backend/app/api/__init__.py` 에 라우터 export
7. `backend/app/main.py` 에 `app.include_router()` 추가
8. `alembic revision --autogenerate -m "add_<table_name>"` 으로 migration 생성
9. 생성된 migration 파일 검토 후 적용

### 4.4 env.py 구조

`backend/alembic/env.py`는 다음을 담당한다:

- `.env` 파일에서 `CONN_URL` 자동 로드 (`dotenv`)
- 모든 SQLModel 모델 import (메타데이터 등록)
- `CONN_URL`의 `%` 문자가 configparser와 충돌하므로 `create_engine`으로 직접 엔진 생성

**모델을 추가/삭제할 때 반드시 이 파일의 import를 업데이트해야 한다.**

### 4.5 autogenerate 주의사항

Alembic autogenerate는 다음을 감지하지 **못한다**:

- 테이블/컬럼 이름 변경 (drop + create로 인식)
- 데이터 마이그레이션 (INSERT, UPDATE 등)
- MySQL의 ENUM 타입 변경
- 인덱스 이름만 변경

이런 경우 빈 revision을 생성하고 수동으로 작성한다:

```bash
.venv/bin/alembic revision -m "rename_column_example"
```

```python
def upgrade() -> None:
    op.alter_column('user', 'old_name', new_column_name='new_name')

def downgrade() -> None:
    op.alter_column('user', 'new_name', new_column_name='old_name')
```

## 5. 세션 관리

### 5.1 기본 패턴

FastAPI 의존성 주입으로 세션을 관리한다 (`app/core/mydb.py`):

```python
def get_session():
    with Session(engine) as session:
        yield session
```

### 5.2 CRUD에서의 세션 사용

```python
# 올바른 패턴
def create_item(session: Session, item: MyModel) -> MyModel:
    session.add(item)
    session.commit()
    session.refresh(item)   # commit 후 반드시 refresh
    return item
```

**주의:**
- `session.commit()` 후 세션의 모든 객체가 만료됨
- 관련 객체까지 포함하여 `session.refresh()` 호출 필요
- 한 함수 = 한 트랜잭션 원칙

```python
# 관련 객체가 있을 때
team = Team(...)
session.add(team)
session.commit()
session.refresh(team)      # team refresh

hero = Hero(team=team)
session.add(hero)
session.commit()
session.refresh(hero)
session.refresh(team)       # team도 다시 refresh 필요!
```

## 6. 데이터 안전 규칙

### 6.1 Soft Delete

모든 핵심 테이블은 `deleted_at` 컬럼으로 Soft Delete를 적용한다.

```python
# 삭제 시 — deleted_at에 현재 시각 기록
item.deleted_at = datetime.now()
session.commit()

# 조회 시 — deleted_at이 NULL인 레코드만 조회
statement = select(MyModel).where(MyModel.deleted_at == None)
```

물리적 삭제(`session.delete()`)는 테스트 데이터 정리 등 예외적 경우에만 사용한다.

### 6.2 센터별 데이터 격리

모든 API에서 센터 ID 기반 권한 검증이 필수다:

```python
# 선생님 — 자기 센터 데이터만 접근
statement = select(ClientInfo).where(
    ClientInfo.consultant == current_user.username
)

# 센터장 — 자기 센터 데이터만 관리
statement = select(User).where(
    User.center_username == current_user.username
)
```

### 6.3 테이블 제거 절차

테이블이나 컬럼을 제거할 때는 다음 안전 절차를 따른다:

```
1. 사용 현황 분석 (코드 기반 grep)
2. 데이터 매핑 검증 (현행 테이블에 데이터 존재 확인)
3. 코드에서 참조 제거
4. DB에서 rename (suffix: _deprecated_YYYYMMDD)
5. 1~2주 운영 관찰
6. 이상 없으면 최종 DROP
```

**절대로 바로 DROP하지 않는다.** rename → 관찰 → drop 순서를 지킨다.

## 7. 백업 및 복원

### 7.1 백업

```bash
# 스키마만
mysqldump -h <host> -u <user> -p --no-data <database> > schema_$(date +%Y%m%d).sql

# 전체 (스키마 + 데이터)
mysqldump -h <host> -u <user> -p <database> > full_backup_$(date +%Y%m%d).sql

# 특정 테이블만
mysqldump -h <host> -u <user> -p <database> user centerinfo clientinfo > tables_$(date +%Y%m%d).sql
```

### 7.2 복원

```bash
mysql -h <host> -u <user> -p <database> < backup_file.sql
```

### 7.3 백업 필수 시점

- Alembic migration 적용 전
- 테이블 구조 변경 전
- 데이터 대량 수정/삭제 전
- 프로덕션 배포 전

## 9. 트러블슈팅

### Alembic 연결 실패

```
sqlalchemy.exc.OperationalError: unable to open database file
```

→ `CONN_URL`이 설정되지 않았거나 `.env` 파일 경로가 잘못됨. `alembic/env.py`의 `load_dotenv` 경로 확인.

### configparser interpolation 에러

```
ValueError: invalid interpolation syntax in '...' at position N
```

→ `CONN_URL`에 `%` 문자가 포함된 경우 발생. `env.py`에서 `config.set_main_option()` 대신 `create_engine()`을 직접 사용하여 우회.

### autogenerate가 변경을 감지하지 못함

→ `alembic/env.py`에서 새 모델의 import가 누락됨. `target_metadata`에 등록되려면 모든 모델이 import되어야 한다.

### 순환 import 에러

```
ImportError: cannot import name 'X' from partially initialized module
```

→ 관계를 가진 모델을 다른 파일에 분리했을 때 발생. 관계가 있는 모델은 반드시 같은 파일에 둔다.

### migration 충돌 (다중 head)

```bash
# 충돌 확인
.venv/bin/alembic heads

# 병합 migration 생성
.venv/bin/alembic merge -m "merge_heads" <head1> <head2>
```
