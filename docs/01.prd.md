## 1. 프로젝트 개요

**프로젝트명**: ITokTok (아이톡톡)
**도메인**: 아동 심리 상담센터 운영 관리
**플랫폼**: 웹 애플리케이션 (데스크톱 전용, 모바일은 별도 Expo 앱으로 구축 예정)
**기술 스택**: FastAPI (백엔드) + Vue 3 (프론트엔드) + MySQL/MariaDB (데이터베이스)

아이톡톡은 아동 심리 상담센터의 전반적인 운영을 디지털화하는 종합 관리 플랫폼이다. 센터장, 선생님(상담사), 최고관리자가 각자의 역할에 맞는 권한 범위 내에서 내담자 관리, 상담 일정 조율, 상담 기록 작성, 바우처 처리 등 일상적인 센터 운영 업무를 하나의 시스템에서 처리할 수 있도록 한다.

## 2. 배경 및 목적

### 배경

아동 심리 상담센터는 특수한 운영 특성을 가진다.

- 다수의 내담자(아동)와 보호자의 정보를 민감하게 관리해야 한다.
- 선생님별 상담 일정이 복잡하게 얽혀 있어 일정 충돌 방지가 중요하다.
- 국가 복지 바우처(발달재활서비스 등)를 처리하는 행정 업무가 병행된다.
- 상담 기록은 법적·윤리적 보관 의무가 있으며 외부 유출을 엄격히 제한해야 한다.

기존에는 종이 기록, 개인 엑셀 파일, 개별 앱을 혼용하는 방식으로 운영되어 데이터 일관성 부족, 정보 유실, 업무 비효율이 반복되었다.

### 목적

| 목적 | 설명 |
|------|------|
| 업무 통합 | 일정·기록·바우처·공지를 하나의 플랫폼에서 처리 |
| 데이터 안전성 | 센터별 데이터 격리, 역할 기반 접근 제어(RBAC) 적용 |
| 운영 가시성 | 월간/주간/일간 캘린더 뷰로 센터 운영 현황 즉시 파악 |
| 확장성 | 다수 센터를 단일 플랫폼에서 독립적으로 운영 가능한 멀티테넌트 구조 |

## 3. 사용자 페르소나

### 페르소나 1: 최고관리자 (SuperAdmin)

| 항목 | 내용 |
|------|------|
| 식별 조건 | `is_superuser == 1` |
| 역할 | 전체 시스템 운영 관리자 |
| 주요 목표 | 플랫폼에 등록된 모든 센터와 사용자 현황 파악, 장애 대응, 시스템 설정 관리 |
| 주요 행동 | 전체 센터 목록 조회, 모든 사용자 계정 관리, 시스템 공지사항 발행 |
| 접근 범위 | 전체 데이터 무제한 접근 |
| 고충 | 개별 센터의 데이터 상태를 파악하기 어려움, 사용자 권한 오남용 모니터링 필요 |

### 페르소나 2: 센터장 (Center Director)

| 항목 | 내용 |
|------|------|
| 식별 조건 | `user_type == 1` |
| 역할 | 센터 운영 총괄 책임자 |
| 주요 목표 | 소속 선생님과 내담자를 효율적으로 관리하고 센터 운영 현황을 실시간으로 파악 |
| 주요 행동 | 센터 생성 및 설정, 선생님 계정 생성, 내담자 등록, 전체 일정 조회, 바우처 관리 |
| 접근 범위 | 자신이 운영하는 센터의 데이터 전체 |
| 고충 | 선생님별 일정 충돌 파악, 바우처 만료·소진 현황 추적, 복수의 행정 업무 병행 |

### 페르소나 3: 선생님 (Therapist / Teacher)

| 항목 | 내용 |
|------|------|
| 식별 조건 | 기본 사용자 (`user_type` 기본값) |
| 역할 | 실제 상담·치료를 담당하는 상담사 |
| 주요 목표 | 담당 내담자의 일정과 상담 기록을 빠르게 등록·조회 |
| 주요 행동 | 자신의 일정 확인·등록, 상담 후 기록 작성, 담당 내담자 정보 조회 |
| 접근 범위 | 자신이 담당하는 내담자 및 본인의 일정 데이터 |
| 고충 | 상담 후 빠른 기록 작성 필요, 반복 일정 등록 번거로움, 다음 일정 현황 즉시 파악 필요 |

## 4. 핵심 기능 요구사항

### 4.1 인증 (Authentication)

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| AUTH-01 | 이메일/사용자명 + 비밀번호로 로그인 (OAuth2 Password Flow) | 필수 |
| AUTH-02 | JWT 액세스 토큰 발급 및 만료 처리 | 필수 |
| AUTH-03 | 비밀번호 분실 시 재설정 기능 | 필수 |
| AUTH-04 | 401 응답 시 자동 로그아웃 및 로그인 페이지 리다이렉트 | 필수 |
| AUTH-05 | 센터장 자가 회원가입 (가입 후 센터 생성 흐름) | 필수 |

### 4.2 일정 관리 (Schedule Management)

일정은 두 계층 구조로 관리된다.

- **Schedule (반복 일정 묶음)**: 특정 내담자-선생님-프로그램 조합의 기간과 반복 규칙을 정의
- **ScheduleList (개별 일정 항목)**: 반복 일정에서 생성된 날짜별 단건 일정, 상태 관리 포함

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| SCH-01 | 일정 생성 시 반복 유형 지정: 매일(1), 매주(2), 매월(3) | 필수 |
| SCH-02 | 매주 반복 시 요일 개별 지정 (월~일 각각 ON/OFF) | 필수 |
| SCH-03 | 일정 유형 구분: 재활(1), 상담/평가(2), 기타(3) | 필수 |
| SCH-04 | 개별 일정 상태 관리: 예약(1), 완료(2), 취소(3), 노쇼(4), 보류(5) | 필수 |
| SCH-05 | 수정 범위 선택: 이번 일정만 변경(single) / 이후 반복 일정 모두 변경(all) | 필수 |
| SCH-06 | 월간 캘린더 뷰 (MonthlyView) | 필수 |
| SCH-07 | 주간 캘린더 뷰 (WeeklyView) | 필수 |
| SCH-08 | 일간 상세 뷰 슬라이딩 패널 (DailyViewSliding) | 필수 |
| SCH-09 | 일정 등록/수정 슬라이딩 폼 (ScheduleFormSliding) | 필수 |
| SCH-10 | 센터장은 센터 전체 선생님 일정 조회 가능 | 필수 |
| SCH-11 | 선생님은 자신의 일정만 조회·수정 가능 | 필수 |
| SCH-12 | 일정에 메모 첨부 기능 | 선택 |

### 4.3 내담자 관리 (Client Management)

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| CLI-01 | 내담자 등록: 이름, 전화번호, 생년월일, 성별, 주소, 이메일, 가족 정보 | 필수 |
| CLI-02 | 내담자 상담 상태 관리 (`consultant_status`) | 필수 |
| CLI-03 | 내담자에 담당 선생님 지정 (`consultant`) | 필수 |
| CLI-04 | 내담자 태그 기능 (검색·분류용) | 선택 |
| CLI-05 | 내담자 메모 기능 | 선택 |
| CLI-06 | 내담자 상담 경로(consultation_path) 기록 | 선택 |
| CLI-07 | 내담자 목록 검색 및 필터링 | 필수 |
| CLI-08 | 내담자 등록/수정 슬라이딩 폼 (ClientFormSliding) | 필수 |
| CLI-09 | 내담자 소프트 삭제 (deleted_at 기반) | 필수 |
| CLI-10 | 센터장은 센터 전체 내담자 조회, 선생님은 담당 내담자만 조회 | 필수 |

## 5. 비기능 요구사항

### 5.1 보안

| 항목 | 요구사항 |
|------|----------|
| 인증 | JWT 기반 인증, bcrypt 비밀번호 해싱 |
| 권한 | 역할 기반 접근 제어 (RBAC): 최고관리자 > 센터장 > 선생님 |
| 데이터 격리 | 모든 API에서 센터 ID 기반 필터링 필수, 타 센터 데이터 접근 불가 |
| 전송 보안 | 프로덕션 환경 HTTPS 필수 (민감한 개인정보 포함) |
| 민감 정보 | 내담자 개인정보는 개인정보보호법 준수, 불필요한 로그 출력 금지 |
| 토큰 관리 | 액세스 토큰 만료 시 자동 로그아웃, 토큰 노출 최소화 |

### 5.2 성능

| 항목 | 요구사항 |
|------|----------|
| 응답 시간 | 일반 API 응답 < 500ms (p95) |
| 페이지네이션 | 목록 API는 `fastapi-pagination` 기반 페이지네이션 필수 (무제한 조회 금지) |
| 정적 자원 | Nginx 정적 파일 캐싱으로 프론트엔드 로딩 최적화 |
| 코드 스플리팅 | Vue Router lazy loading으로 초기 번들 크기 최소화 |

### 5.3 확장성

| 항목 | 요구사항 |
|------|----------|
| 멀티테넌트 | 단일 플랫폼에서 다수의 독립 센터 운영 지원 |
| 컨테이너화 | Docker Compose 기반 배포, 수평 확장 가능한 구조 |
| 모바일 확장 | 현 REST API는 향후 Expo 모바일 앱과 동일 엔드포인트 공유 설계 |
| DB 분리 | MySQL/MariaDB(주) + MongoDB(보조)의 이중 데이터베이스 구조 지원 |

### 5.4 유지보수성

| 항목 | 요구사항 |
|------|----------|
| 레이어 분리 | `models/` → `crud/` → `api/` → `schemas/` 계층 구조 준수 |
| 소프트 삭제 | 주요 엔티티는 `deleted_at` 필드로 소프트 삭제 처리 |
| 감사 로그 | 모든 엔티티에 `created_by`, `updated_by`, `created_at`, `updated_at` 기록 |
| 코드 품질 | 백엔드: black/ruff 포맷팅, 프론트엔드: ESLint + Prettier |

### 5.5 가용성

| 항목 | 요구사항 |
|------|----------|
| 배포 방식 | Docker Compose 기반 단일 서버 배포 (초기) |
| 백업 | 외부 MySQL 서버의 정기 백업 정책 적용 |
| 오류 처리 | 모든 API 오류는 명확한 HTTP 상태 코드와 메시지 반환 |

## 8. 제약사항 및 가정

### 제약사항

| 구분 | 내용 |
|------|------|
| 플랫폼 | 데스크톱 웹 브라우저 전용 (반응형 미지원, 모바일은 Expo 별도 구축) |
| 데이터베이스 | 외부 MySQL/MariaDB 서버 사용 (Docker 내장 DB 미사용) |
| 인증 방식 | JWT 기반 단일 토큰, 리프레시 토큰 미구현 (초기 버전) |
| 배포 환경 | Docker Compose 단일 서버 배포 (쿠버네티스 등 오케스트레이션 미적용) |
| 아키텍처 | Mac M1/M2 빌드 시 `linux/amd64` 플랫폼 명시 필요 |

### 가정

| 구분 | 내용 |
|------|------|
| 사용자 환경 | 센터장 및 선생님은 데스크톱/노트북 환경에서 운영 업무 처리 |
| 네트워크 | 시스템 사용 환경에서 인터넷 연결 상시 가능 |
| 데이터 소유 | 각 센터의 데이터는 해당 센터장에게 귀속됨 |
| 초기 설정 | 센터 최초 운영 시 센터장이 직접 선생님·프로그램·내담자를 등록 |
| 법적 준수 | 개인정보보호법에 따른 내담자 정보 처리는 센터 운영자 책임으로 가정 |
| 바우처 연동 | 외부 복지부 바우처 시스템과의 실시간 API 연동은 미포함 (수동 입력 방식) |
| 알림 기능 | 이메일·SMS·푸시 알림은 초기 버전 미포함, 향후 확장 예정 |
