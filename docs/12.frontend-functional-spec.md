## 1. 개요

### 1.1 문서 목적

본 문서는 ITokTok 프론트엔드(데스크톱 웹)의 **현행 구현 기능**을 정의하고, 다음 4가지 용도의 기반 문서로 활용한다:

| 용도 | 활용 방식 |
|------|----------|
| **기능 최적화** | 각 기능의 현행 동작을 기준으로 개선점 도출 |
| **누락/개선 기능** | 비즈니스 룰 대비 미구현 기능 식별 |
| **모바일 버전 기능 정의** | 모바일 대응 태그(필수/선택/제외)로 Expo 앱 기능 범위 결정 |
| **통합 e2e 테스트** | 기능별 정상/예외 시나리오를 테스트 케이스 기준으로 활용 |

### 1.2 용어 정의

| 용어 | 정의 |
|------|------|
| **센터** (Center) | 아동 심리 상담센터 단위. 데이터 격리의 기본 단위 |
| **센터장** (Center Director) | 센터를 생성하고 운영하는 실질적 서비스 고객 |
| **선생님** (Teacher) | 센터에 소속되어 상담을 수행하는 상담사 |
| **내담자** (Client) | 상담을 받는 아동/대상자 |
| **프로그램** (Program) | 상담에 사용되는 프로그램/교구 |
| **일정** (Schedule) | 상담 일정. 선생님-내담자-프로그램의 조합 |
| **상담상태** | 내담자의 상담 진행 상태: 상담진행(1), 상담보류(2), 상담종결(3) |
| **슬라이딩 패널** | 화면 우측에서 슬라이드인되는 상세 정보/폼 영역 |

## 2. 역할 체계

### 2.1 역할 정의

| 역할 | 식별 조건 | 등록 방식 | 핵심 역할 |
|------|----------|----------|----------|
| **최고관리자** (Superuser) | `is_superuser == 1` | 시스템 지정 | 시스템 전체 운영/관리 |
| **센터장** (Center Director) | `user_type == 1` | 자가 회원가입 (`/signup`) | 실질적 서비스 고객. 센터 생성, 선생님/내담자 관리 |
| **선생님** (Teacher) | 기본값 (위 조건 해당 없음) | 센터장이 생성 | 담당 내담자 상담 수행, 일정/기록 관리 |

### 2.2 역할별 데이터 범위

| 데이터 | 최고관리자 | 센터장 | 선생님 |
|--------|----------|--------|--------|
| 센터 정보 | 전체 센터 | 자기 센터만 | 소속 센터 (조회만) |
| 사용자(선생님) | 전체 사용자 | 자기 센터 소속 선생님 | 자기 정보만 |
| 내담자 | 전체 내담자 | 자기 센터에 등록된 내담자 | 자신에게 배정된 내담자 |
| 일정 | 전체 일정 | 센터 소속 선생님들의 일정 | 자기 일정만 |
| 프로그램 | 전체 프로그램 | 센터 프로그램 | 센터 프로그램 (조회만) |

### 2.3 역할별 기능 권한 매트릭스

> C = Create, R = Read, U = Update, D = Delete

| 기능 도메인 | 최고관리자 | 센터장 | 선생님 |
|------------|----------|--------|--------|
| 센터 관리 | CRUD | RU (자기 센터) | - |
| 사용자(선생님) 관리 | CRUD | CRUD (소속 선생님) | RU (자기 정보) |
| 내담자 관리 | R (전체) | CRUD (센터 내담자) | CRUD (담당 내담자) |
| 내담자 상담상태 변경 | R | U | U (담당만) |
| 내담자 상담자 배정 | - | U | - |
| 일정 관리 | R (전체) | CRUD (센터 일정) | CRUD (자기 일정) |
| 일정 드래그앤드롭 이동 | R | U | U (자기 일정) |
| 프로그램 관리 | CRUD | CRUD (센터) | R |
| 설정/내정보 | RU | RU | RU |

### 2.4 역할별 검색 범위

| 검색 대상 | 최고관리자 | 센터장 | 선생님 |
|----------|----------|--------|--------|
| 사용자 목록/검색 | 모든 센터의 모든 사용자 | 자기 센터 소속 선생님 | 불가 |
| 내담자 목록/검색 | 모든 센터의 모든 내담자 | 자기 센터 내담자 | 자기 담당 내담자 |
| 일정 캘린더 필터 | 전체 선생님 필터 | 센터 소속 선생님 필터 | 자기 일정만 (필터 불가) |
| 프로그램 검색 | 전체 프로그램 | 센터 프로그램 | 센터 프로그램 |

## 3. 기능 도메인

### 3.1 인증/계정 (Auth)

#### FN-AUTH-001: 로그인

| 항목 | 내용 |
|------|------|
| **기능명** | 로그인 |
| **설명** | 사용자 인증 후 JWT 토큰을 발급받아 서비스에 접근 |
| **접근 권한** | 모든 역할 (비인증 상태에서 접근) |
| **역할별 동작** | 동일. 로그인 후 역할 자동 판별 (`is_superuser` → admin, `user_type==1` → center, 기타 → teacher) |
| **데이터 범위** | 해당 없음 |
| **관련 화면** | `LoginView.vue` |
| **API 매핑** | `POST /auth/login` (form-urlencoded) → `GET /users/me` |
| **입력 필드** | |

| 필드명 | 타입 | 필수 | 유효성 규칙 |
|--------|------|------|------------|
| username | string | Y | 필수 입력 |
| password | string | Y | 최소 8자 |

| 항목 | 내용 |
|------|------|
| **출력/결과** | 성공: JWT 토큰 저장 → `/admin/weekly`로 리다이렉트. 실패: 에러 메시지 표시 |
| **비즈니스 룰** | - 토큰은 localStorage에 저장 (`VITE_TOKEN_KEY`)<br>- 사용자 정보는 Pinia store + localStorage에 영속화<br>- 역할은 `userStore.setUserInfo()`에서 자동 결정 |
| **정상 시나리오** | 1. username/password 입력 → 2. 로그인 버튼 클릭 → 3. 토큰 발급 → 4. `/users/me`로 사용자 정보 조회 → 5. 역할 결정 및 저장 → 6. `/admin/weekly`로 이동 |
| **예외 시나리오** | - 잘못된 자격 증명: 에러 메시지 표시, 페이지 유지<br>- 유효성 검증 실패: 필드별 에러 메시지 (VeeValidate)<br>- 서버 미응답: 네트워크 에러 처리 |
| **e2e 테스트 힌트** | - 정상 로그인 후 `/admin/weekly` 도달 확인<br>- 잘못된 비밀번호로 로그인 실패 확인<br>- 빈 필드 제출 시 유효성 에러 확인<br>- 로그인 후 localStorage에 토큰 존재 확인<br>- 역할별(superuser/center/teacher) 로그인 후 역할 판별 확인 |
| **모바일 대응** | **필수** — 모든 사용자의 진입점 |

#### FN-AUTH-002: 로그아웃

| 항목 | 내용 |
|------|------|
| **기능명** | 로그아웃 |
| **설명** | 현재 세션을 종료하고 인증 정보를 삭제 |
| **접근 권한** | 모든 역할 (인증 상태) |
| **역할별 동작** | 동일 |
| **관련 화면** | 헤더 영역 (`HeaderView.vue`) → 로그아웃 모달 → `LoginView.vue` |
| **API 매핑** | `POST /auth/logout` |
| **입력 필드** | 없음 |
| **출력/결과** | 토큰 삭제, userStore 초기화, `/login`으로 리다이렉트 |
| **비즈니스 룰** | - localStorage의 토큰 및 userStore 데이터 삭제<br>- 로그아웃 모달(`showLogoutModal`) 표시 후 확인 시 처리 |
| **정상 시나리오** | 1. 로그아웃 버튼 클릭 → 2. 로그아웃 모달 표시 → 3. 확인 → 4. API 호출 → 5. 토큰/상태 삭제 → 6. `/login`으로 이동 |
| **예외 시나리오** | - API 호출 실패해도 클라이언트측 토큰 삭제는 실행<br>- 401 응답 인터셉터에 의한 자동 로그아웃 |
| **e2e 테스트 힌트** | - 로그아웃 후 localStorage 토큰 삭제 확인<br>- 로그아웃 후 인증 필요 페이지 접근 시 `/login` 리다이렉트 확인<br>- 401 응답 시 자동 로그아웃 확인 |
| **모바일 대응** | **필수** |

#### FN-AUTH-003: 회원가입

| 항목 | 내용 |
|------|------|
| **기능명** | 회원가입 (센터장 등록) |

| **API 매핑** | `GET /signup/check-username` → `POST /signup` |
| **입력 필드** | |

| 필드명 | 타입 | 필수 | 유효성 규칙 |
|--------|------|------|------------|

| password | string | Y | 최소 8자 |
| center_username | string | Y | 자동 설정 (username 기반) |

| 항목 | 내용 |
|------|------|

| **정상 시나리오** | 1. 정보 입력 → 2. username 중복 확인 → 3. 회원가입 버튼 → 4. 계정 생성 → 5. 로그인 페이지로 이동 |
| **예외 시나리오** | - username 중복: 실시간 에러 표시<br>- 유효성 검증 실패: 필드별 에러<br>- 서버 에러: 에러 메시지 표시 |
| **e2e 테스트 힌트** | - 정상 회원가입 후 로그인 페이지 도달 확인<br>- 중복 username 실시간 에러 확인<br>- 생성된 계정으로 로그인 성공 확인<br>- 등록된 사용자의 `user_type==1` 확인 |
| **모바일 대응** | **필수** — 신규 센터장 온보딩 |

| 반복 타입 | 동작 | 예시 |
|----------|------|------|

**보안 룰:**
- 일정 생성 시 `current_user` 인증 필수
- 타 센터 선생님 지정 불가 (센터별 데이터 격리)

**수정 범위 (update_range):**

| 범위 | 값 | 동작 |
|------|-----|------|
| **해당 일정만** | `single` | 선택한 ScheduleList 1건만 수정. 부모 Schedule은 변경 없음. 해당 건의 시간/선생님/내담자/프로그램 개별 변경 가능. |
| **이후 모든 일정** | `all` | 부모 Schedule 업데이트 → 선택 날짜 이후 ScheduleList 삭제 → 새 반복 규칙으로 재생성. 선택 날짜 이전 일정은 보존. |

**보안 룰:**
- `check_schedule_access()`: 소유자(teacher_username) / 센터장(같은 센터) / 최고관리자만 수정 허용
- 타 센터 사용자 수정 시도 시 403 반환

| 항목 | 내용 |
|------|------|
