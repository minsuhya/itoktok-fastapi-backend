## 1. 개요

### 1.1 목적

본 문서는 `docs/12.frontend-functional-spec.md`에 정의된 45개 기능, 32개 e2e 시나리오를 기반으로, **OpenChrome MCP 플러그인을 통한 실제 브라우저 자동화 테스트**의 실행 계획을 정의한다.

### 1.2 테스트 범위

| 범위                   | 포함                                       | 제외                                  |
| ---------------------- | ------------------------------------------ | ------------------------------------- |
| **역할별 통합 플로우** | 센터장, 선생님, 최고관리자                 | -                                     |
| **기능 도메인**        | 인증, 사용자, 내담자, 일정, 프로그램, 공통 | 센터 관리 (전용 화면 미존재)          |
| **테스트 유형**        | 기능 검증, 역할별 데이터 격리, UI 상호작용 | 성능 테스트, 보안 테스트, 부하 테스트 |
| **브라우저**           | Chrome (OpenChrome CDP)                    | 크로스 브라우저                       |

### 1.3 기능 정의서 대비 커버리지

| 도메인        | 정의서 기능 수 | e2e 시나리오 수 | 커버리지       |
| ------------- | -------------- | --------------- | -------------- |
| 인증/계정     | 6              | 9               | 100%           |
| 사용자 관리   | 6              | 5               | 83%            |
| 내담자 관리   | 7              | 6               | 86%            |
| 일정 관리     | 9              | 10              | 100%           |
| 프로그램 관리 | 4              | 3               | 75%            |
| 센터 관리     | 4              | 0               | 0% (화면 없음) |
| 설정/내정보   | 2              | 0 → 2 추가      | 100%           |
| 공통 기능     | 7              | 6               | 86%            |
| **합계**      | **45**         | **41**          | **91%**        |

### 1.4 기능 정의서 검토 결과

기능 정의서(`docs/12.frontend-functional-spec.md`)를 e2e 테스트 관점에서 검토한 결과:

**강점:**

- 45개 기능에 32개 e2e 시나리오 힌트가 매핑되어 커버리지 기반이 확보됨
- 역할별(최고관리자/센터장/선생님) 데이터 격리 시나리오 5개 별도 정의
- 정상/예외 시나리오가 분리되어 테스트 케이스 변환 용이
- 모바일 대응 태그로 테스트 우선순위 판단 근거 제공

**보완 사항 (본 문서에서 해소):**

| 영역            | 정의서 현황   | 본 문서에서 보완                        |
| --------------- | ------------- | --------------------------------------- |
| 테스트 데이터   | 미정의        | 4절: 역할별 계정 및 시드 데이터 정의    |
| 테스트 URL      | 미명시        | 2절: 환경별 URL 정의                    |
| 시나리오 의존성 | 미정의        | 5절: 역할별 플로우 순서 및 의존성 정의  |
| 검증 기준       | 텍스트 수준   | 6절: DOM 요소/URL 패턴/텍스트 기반 검증 |
| 센터 관리       | API만 존재    | 범위에서 제외, API 테스트로 전환 권장   |
| 설정/내정보     | e2e 힌트 없음 | 6.7절: 시나리오 추가                    |

### 3.2 플러그인별 역할 상세

#### OpenChrome — 브라우저 자동화 엔진

e2e 테스트의 **실행 레이어**. 실제 브라우저 조작을 담당한다.

| 도구                   | e2e 활용                            | 주요 파라미터                          |
| ---------------------- | ----------------------------------- | -------------------------------------- |
| `navigate`             | 페이지 이동                         | `url`                                  |
| `fill_form`            | 폼 입력 (로그인, 등록 등)           | `tabId`, 필드/값 매핑                  |
| `find`                 | 요소 탐색 (버튼, 목록, 에러 메시지) | `tabId`, `query` (자연어)              |
| `interact`             | 클릭, 드롭다운 선택, 패널 열기      | `tabId`, 요소 참조                     |
| `read_page`            | 페이지 상태 확인                    | `tabId`, `mode` (ax/dom/css), `filter` |
| `inspect`              | 자연어 기반 구조 확인               | `tabId`, `query`, `scope`              |
| `computer(screenshot)` | 시각적 검증용 스크린샷 캡처         | `tabId`, `action: screenshot`          |
| `tabs_create`          | 새 탭 열기 (다중 역할 동시 테스트)  | `url`                                  |
| `tabs_context`         | 현재 열린 탭 정보 확인              | -                                      |
| `wait_for`             | 요소/상태 대기 (비동기 로딩)        | `tabId`, 조건                          |
| `javascript_tool`      | 직접 JS 실행 (localStorage 확인 등) | `tabId`, `code`                        |

**OpenChrome 테스트 패턴:**

```
# 기본 패턴: navigate → interact → verify
navigate(url)
  → fill_form(credentials)
  → interact(submit button)
  → wait_for(page load)
  → read_page(verify expected state)
  → computer(screenshot for evidence)

# 검증 패턴: find → read_page → assert
find("error message elements")
  → read_page(mode: ax, filter: interactive)
  → inspect("verify specific content")

# localStorage 검증 패턴:
javascript_tool("return localStorage.getItem('access_token')")
  → null이면 미인증, 값이면 인증 상태
```

### 3.3 플러그인 조합 패턴

#### 패턴 A: 단일 역할 순차 테스트

```
superpowers:tdd (시나리오 설계)
  → openchrome (순차 실행)
  → visual-verdict (스크린샷 검증)
  → verification-before-completion (완료 게이트)
```

**적합**: 빠른 검증, 단일 역할 회귀 테스트

#### 패턴 B: 역할별 병렬 테스트

```
test-engineer (전략 설계)
  → ultrawork [
      Agent A: openchrome (센터장 플로우),
      Agent B: openchrome (선생님 플로우),
      Agent C: openchrome (관리자 플로우)
    ]
  → verifier (통합 결과 검증)
```

**적합**: 전체 역할 커버리지, 데이터 격리 검증

#### 패턴 C: 자동 복구 루프

```
ultraqa [
  cycle 1: openchrome 실행 → 실패 감지
  cycle 2: debugger 진단 → executor 수정 → 재실행
  cycle 3~5: 반복 (조기 종료 가능)
]
  → octo:staged-review (최종 품질 검증)
```

**적합**: 불안정한 환경, 버그 수정 후 회귀 테스트

## 9. 실패 처리 및 복구

### 9.1 자동 복구 (ultraqa)

```
ultraqa 사이클:
  Cycle 1: 테스트 실행 → 실패 감지
  Cycle 2: architect 진단 → "로그인 폼 셀렉터 변경" 발견
  Cycle 3: executor 수정 → find 쿼리 업데이트 → 재실행
  Cycle 4: 통과 → 완료

  (최대 5사이클, 같은 에러 반복 시 조기 종료)
```

### 9.2 실패 분류

| 분류            | 원인                       | 대응                                       |
| --------------- | -------------------------- | ------------------------------------------ |
| **환경 실패**   | 서버 미기동, DB 미연결     | Phase 0 환경 검증에서 차단                 |
| **데이터 실패** | 시드 데이터 누락/오염      | 데이터 재시드 후 재실행                    |
| **셀렉터 실패** | UI 변경으로 요소 찾기 실패 | `inspect`로 현재 구조 파악 → 쿼리 업데이트 |
| **타이밍 실패** | 비동기 로딩 미완료         | `wait_for` 대기 추가                       |
| **기능 실패**   | 실제 버그 발견             | 리포트 생성 → systematic-debugging 연계    |

### 9.3 실패 리포트 형식
